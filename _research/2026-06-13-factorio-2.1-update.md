## P2. New blueprint-paste API -> retire the marker hack

`on_blueprint_settings_pasted` (added 2.1.0) is the event the marker-injection system was
a workaround for. Verified semantics from game source (`GameEvents.cpp:2954`,
`BlueprintBuilder.cpp:1037`):

> Called when a blueprint entity is pasted over an existing entity or entity ghost.
> Settings, rotation, mirroring, wires may have changed. Raised even if nothing changed.
> NOT raised for upgrades, new entities, or instant ghost revives (those raise
> on_built_entity).
> Params: entity (may be ghost), player_index?, tags? (source blueprint tags; non-ghost
> only -- ghosts read entity.tags), previous_direction?, mirrored (boolean).

This is exactly the "overbuild existing entity to update its staged settings" case. Once
`base >= 2.1`, the whole hacky path can be deleted:
- marker injection: `prepareBlueprintForStagePaste`, `onEntityMarkerBuilt`,
  `handleEntityMarkerBuilt`, `onLastEntityMarkerBuilt`, `revertPreparedBlueprint`,
  `tryFixBlueprint`, the `EntityMarker` prototype, and `currentBlueprintPaste` state.
- the interim bplib path (`onPreBlueprintPastedBplib`, `BplibPasteEvent`,
  `pendingBplibPaste`, `UseBplibForBlueprintPaste` setting, `bplib` dependency).

### Status: DONE (switch + phase-B deletion both complete)

The mod now drives blueprint paste off the native events (no marker injection, no bplib).
All tests pass (1339 passed / 3 skipped after dropping the redundant bplib parametrizations).

What was done (`build-events.ts`, `blueprint-paste.ts`):
- `Events.on_blueprint_settings_pasted` (existing entities pasted over) -> `applyPastedEntity`
  -> `onEntityPossiblyUpdated(entity, stage, previous_direction, player, tags.bp100, items)`
  then `onWiresPossiblyUpdated`.
  - IMPORTANT: must register via `ProtectedEvents`/`Events`, NOT raw `script.on_event` -- the
    latter is silently clobbered by the `Events` lib and the handler never fires.
- `on_built_entity` paste-created entities: when `e.tags` carry `bp100`/items, route through
  the same `applyPastedEntity`; otherwise normal `onEntityCreated` (which already saves wires
  + reads item-request proxies). Native paste connects wires before these events fire.
- Item requests: the native events deliver `tags` but NOT `items`, and the world entity's
  item-request-proxy is empty at event time (editor instant-fulfills + discards it). Solution:
  in `on_pre_build` stamp each blueprint entity's `items` into a `bp100Items` tag
  (`stampBlueprintItemTags`), recover via `getPasteItemRequests(event.tags)`, then strip the
  tag next tick (`RevertPasteItemTagsEvent` DelayedEvent) so the player's blueprint is
  unchanged. This sidesteps blueprint-entity <-> world-position mapping entirely (the engine
  carries the data through attached to the right entity).
- P3 mirroring bug resolved as a side effect: native paste sets `mirroring` directly, so the
  asymmetric `pasteEntity` path (`save-load.ts:184`, only set true never false) is no longer
  hit on paste.

Phase B (DONE): deleted all dead marker/bplib code -- `prepareBlueprintForStagePaste`,
`onEntityMarkerBuilt`/`handleEntityMarkerBuilt`/`onLastEntityMarkerBuilt`,
`revertPreparedBlueprint`, `tryFixBlueprint`, `manuallyConnectWires`, `currentBlueprintPaste`
state, the `EntityMarker` prototype + locale + filter; the bplib path
(`onPreBlueprintPasted(Bplib)`, `processPendingBplibPaste`, `BplibPasteEvent`,
`pendingBplibPaste`, `findPastedEntity`, `calculateTransformedDirection`), the
`UseBplibForBlueprintPaste` setting + locale, `src/declarations/bplib.d.ts`, and the `bplib`
dependency in `src/info.json` (+ `factorio-test.json` test mods). `base` bumped to `2.1.0`
and `factorio_version` to `2.1`. Test parametrizations (`[markers, bplib]` /
`UseBplibForBlueprintPaste` + the now-synchronous `waitForPaste` helper) dropped. A migration
in `blueprint-paste.ts` resets `storage.blueprintPasteState` to drop the stale transient keys.

## P3. Inserter / drill flipping -> native mirroring

2.1.0: "Added pumpjack and burner mining drill flipping", "Inserters can be flipped to
make blueprint flipping reliable." Entities now carry a `mirroring` flag (and blueprint
entities a `mirror` field) instead of faking flips via direction.
- Scripting: `LuaEntity.flip`; `LuaEntityPrototype.use_mirroring`; `mirrored` param on
  `get_item_insert_specification`; `previous_mirroring` on `on_player_rotated_entity`.
- Prototype: `InserterPrototype.use_mirroring`, `MiningDrillPrototype.use_mirroring` +
  `migrate_horizontal_mirroring`, `CraftingMachinePrototype.use_mirroring` (replaced
  `forced_symmetry`).

### Status: DONE (no code change needed; verified by test)

The mod's mirror handling is fully generic, never keyed by entity type, so inserters/drills
get 2.1 mirroring for free:
- `mirror` is a base `BlueprintEntity` field (typed-factorio `concepts.d.ts:7406`).
- Save: `blueprintEntity()` -> `create_blueprint` (`save-load.ts:94`) captures `mirror` for
  any mirrorable proto.
- Apply to existing: `luaEntity.mirroring = value.mirror ?? false` (`save-load.ts:557`).
- Apply to new/paste: `pasteEntity` -> `build_blueprint` carries `value.mirror`.
- World flip: generic `on_player_flipped_entity` -> `luaEntityPossiblyUpdated`
  (`entity-events.ts:60`) -> full re-save.
Direction is orthogonal and always set (`save-load.ts:556`); the old direction-faking
`calculateTransformedDirection` was deleted with the marker hack in P2.

Verified: added `test.each(["inserter","burner-mining-drill"])("mirroring a %s")` in
`blueprint-paste.test.ts` -- flip in stage 1, propagate, then `assertEntityCorrect` (full
save/load round-trip across all stages). Both pass, confirming 2.1's `create_blueprint`
emits `mirror` for these types.

Non-issues (no action):
- `on_player_rotated_entity` ignores `previous_mirroring` (`entity-events.ts:62`): a pure
  flip fires `on_player_flipped_entity` instead (full re-save); the mod does no delta logic
  keyed on prior mirror state.
- AM-only block at `save-load.ts:180-187` sets `mirroring=true` only -- a recipe-path fixup;
  inserters/drills use the symmetric set at `:557`.

Not retested (lower priority, separate from staging round-trip): inserter-with-fluid-energy
flip (2.1.4), splitter flip lanes, underground-belt ghost flip.
